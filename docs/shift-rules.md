## 制約競合パターン

| 競合 | シナリオ | 期待する動作 |
|------|---------|------------|
| R1 vs R14 | 全員が同日に希望休 | R1優先（希望を一部キャンセル）。緩和報告あり |
| R6 vs R14 | 希望反映すると5連勤になる | R14優先でもR6違反は報告する |
| R13 vs 人員構成 | 公休数を守ると日勤が不足する | **R13は緩和禁止**。人員不足日は空欄で対応（R16） |
| R9 vs R13 | 連休付与すると公休超過になる | **R13優先**。連休は部分緩和 |
| R12 vs R11 | 月末土日に代休が発生 | 翌月取得として処理し報告 |
| R5 vs R2 | 当直翌日（明け）に日勤2名が必要 | 明けの人は日勤に入れず、他から補充 |
| R10 vs R14 | 日曜休み均等にすると希望が通らない | R14優先（希望最優先）。R10は後から調整 |

## 月跨ぎの考慮事項

- 前月末が「当」で終わった場合 → 当月1日を「明」に確定する
- 前月末の連勤日数を引き継ぎ、月初に連勤上限を超えないよう調整する
- 前月末が土日の当直で代休が当月に引き継がれた場合 → 当月内に代休を付与（または翌月取得の旨を報告）

## アルゴリズム概要（Phase B: 公休事前配置）

R2/R12 バグ解消のため 2026-04-23 に Phase B を per-staff → per-date 処理に再設計。

### 処理フロー

| フェーズ | 内容 |
|---------|------|
| B-1 | 日曜を1件ずつ走査し、R10（日曜休み均等）を守りながら公休を配置 |
| B-2 | 土曜を1件ずつ走査し、日勤2名確保（coverage check）しながら公休を配置 |
| B-3 | 平日を1件ずつ走査し、日勤2名確保（coverage check）しながら公休を配置 |
| B-4 | スタッフ別に公休不足分を補填（Pass1: coverage-aware、Pass2: selectEvenly 均等分散、Pass3: 順次フォールバック） |

### coverage check の原則

`placeRestForDay(d, required, maxRest, sortKey)` は「その日に休みを置いても残りの日勤候補が `required` 名以上いる場合のみ配置」する。これにより公休配置が日勤人員を潰さないことを保証する。

### STEP 5 / STEP 6 の容量保護

- **STEP 5（日勤割り当て前）**: 公休残が残っているスタッフを日勤候補から除外し、R13 スロットを確保
- **STEP 6（代休配置）**: 代休を置いた後も残り空きスロット ≥ 必要公休数 の場合のみ代休を配置し、公休スロットを守る
